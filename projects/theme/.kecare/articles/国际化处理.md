# 国际化处理

每一篇用心撰写的文章都值得被更多人看见，值得跨越语言的界限，被全世界的读者阅读。Kecare 为你提供了强大的国际化（i18n）支持，让你的内容能够轻松触达全球受众。

---

## 为什么需要国际化

### 扩大受众范围

- 触达不同语言背景的读者
- 提升文章的国际影响力
- 增加博客的访问量

### 提升用户体验

- 读者可以用母语阅读
- 减少语言障碍带来的理解困难
- 提供更友好的阅读体验

### SEO 优化

- 多语言内容有助于搜索引擎收录
- 提高在不同地区搜索结果中的排名

## 快速开始

### 步骤一：在文章中启用翻译

在文章的 Front Matter 中添加 `translate` 字段：

```yaml
---
title: 我的第一篇文章
translate: ['zh-CN', 'en-US', 'ja-JP']
---
```

> ⚠️ **重要**：第一个语言代码必须是文章的**原始语言**。

### 步骤二：创建配置文件

在 `.kecare` 目录下创建 `kecare.config.ts` 文件。

### 步骤三：配置翻译参数

```ts
// .kecare/kecare.config.ts
export async function kecareConfig() {
    return {
        llm: {
            model: 'deepseek-chat',
            apiKey: 'your-api-key',
            apiBaseUrl: 'https://api.deepseek.com/v1',
            prompt: ''  // 可选：额外的翻译提示词
        }
    };
}
```

### 步骤四：运行生成器

```bash
kecare gen .
```

Kecare 会自动翻译文章并生成对应语言版本。

## 配置翻译功能

### 配置项说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `model` | `string` | 是 | LLM 模型名称 |
| `apiKey` | `string` | 是 | API 密钥 |
| `apiBaseUrl` | `string` | 是 | API 服务地址 |
| `prompt` | `string` | 否 | 自定义翻译提示词 |

### 内置系统提示词

Kecare 内置了优化的翻译提示词：

```text
你是一个很好用的翻译助手，请帮我将下面的文章翻译成目标语言，
注意：
- 不要增删任何 HTML 标签/属性和 Markdown 标签/属性
- 如果是代码块的内容，请不要翻译，只翻译文本节点
- 如果是代码内容，请不要翻译，原样放回
- 保留链接 href、图片 src、id、class 不动
- 代码块占位符（如 __CODE_BLOCK_0__）不要翻译，原样保留
```

### 自定义提示词

你可以通过 `prompt` 字段添加额外的翻译指令：

```ts
export async function kecareConfig() {
    return {
        llm: {
            model: 'deepseek-chat',
            apiKey: 'your-api-key',
            apiBaseUrl: 'https://api.deepseek.com/v1',
            prompt: '请使用正式、专业的语气进行翻译。'
        }
    };
}
```

### 常用语言代码

| 语言 | 代码 |
|------|------|

| 日语 | `ja-JP` |
| 韩语 | `ko-KR` |
| 法语 | `fr-FR` |
| 德语 | `de-DE` |
| 西班牙语 | `es-ES` |
| 俄语 | `ru-RU` |
| 葡萄牙语（巴西） | `pt-BR` |
| 阿拉伯语 | `ar-SA` |

### 语言代码格式

语言代码遵循 [BCP 47](https://www.rfc-editor.org/rfc/bcp/bcp47.txt) 标准：

```
语言代码-国家/地区代码
```

例如：
- `zh-CN`：中文-中国
- `en-US`：英语-美国
- `ja-JP`：日语-日本

## Token 优化策略

AI 翻译的 Token 成本可能较高，Kecare 提供了多重优化措施帮助你节省成本：

### 分段翻译

将 Markdown 文章智能分段，逐段提交给 AI 翻译：

```
文章 → 段落1 → 翻译 → 结果1
      → 段落2 → 翻译 → 结果2
      → 段落3 → 翻译 → 结果3
```

**优势**：
- 减少单次请求的 Token 数量
- 降低 API 调用失败的风险
- 便于并行处理

### 缓存机制

翻译结果自动缓存在 `.kecare/.tmp` 目录下：

```txt
.kecare/
└── .tmp/
    ├── en-US/
    │   └── abc123/
    │       └── translations.json
    └── ja-JP/
        └── abc123/
            └── translations.json
```

**优势**：
- 相同内容无需重复翻译
- 降低 API 调用次数

### 增量更新

下次构建时，系统自动检测变更：

```
检测文章变化 → 识别修改段落 → 仅翻译变更部分 → 合并结果
```

**优势**：
- 只翻译修改的内容
- 大幅节省 Token 消耗
- 提高构建效率

## 常见问题

### Q: 翻译质量不理想怎么办？

A: 可以尝试以下方法：
- 在 `prompt` 中添加翻译风格要求
- 更换更强大的模型
- 对关键段落进行人工校对

### Q: 支持哪些翻译服务？

A: Kecare 支持任何兼容 OpenAI API 格式的服务
### Q: 翻译结果存储在哪里？

A: 翻译结果缓存在 `.kecare/.tmp` 目录下，建议在 `.gitignore` 中忽略该目录。

### Q: 如何清除翻译缓存？

A: 删除 `.kecare/.tmp` 目录即可：

```bash
rm -rf .kecare/.tmp
```
